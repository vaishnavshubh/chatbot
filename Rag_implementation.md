# RAG implementation ‚Äî Financial Literacy Chatbot

This document describes the **retrieval-augmented generation (RAG)** layer added to the project: what was built, where it lives, how it affects the chatbot, and how to extend or disable it.

---

## 1. What problem RAG solves here

The chatbot already uses **phase skills** (`md/skills/...`) to control *behavior* (what to ask, how to extract JSON). RAG adds a **curated knowledge base** of short, reviewable text chunks so that **Phase 3 ‚Äî Evidence Intake** and **Phase 4 ‚Äî Plan Generation** can ground explanations in **your team‚Äôs excerpts**, not only in the model‚Äôs general knowledge.

RAG does **not** replace the orchestrator, state schema, or analyzer. It **augments** the Speaker with extra **system** context during both Phase 3 (evidence intake) and Phase 4 (plan generation).

---

## 2. Architecture (how it fits)

| Layer | Role |
|--------|------|
| **Orchestrator** | Still owns phases, validation, and when to call the Speaker. |
| **Analyzer** | Unchanged; RAG does not feed the analyzer (keeps JSON extraction stable). |
| **Speaker** | Receives an optional **third system message**: formatted retrieved excerpts. |
| **RAG** | Two backends: (1) JSONL keyword retrieval, (2) Chroma vector retrieval from hierarchical PDF chunks. |

**When RAG runs:** When the session is in **Phase 3** (evidence intake) or **Phase 4** (plan generation). Other phases behave as before.

**When RAG does not run:** If `RAG_ENABLED` is off, if selected backend data is missing/empty, or if retrieval fails (then the plan is generated as before, without excerpts).

## 3. Files and directories

| Path | Purpose |
|------|---------|
| `data/rag/*.md` | **Source** Markdown. Filename (without `.md`) = **`topic`**, must match `goal.primary_goal` (e.g. `credit_management.md`). Use `##` headings to split into chunks. |
| `data/rag/README.md` | Short instructions for authors. |
| `data/rag_index/chunks.jsonl` | **Built index** ‚Äî one JSON object per line: `id`, `text`, `source`, `topic`, `heading`. Generated by ingest; safe to commit so the app works without rerunning the script. |
| `app/rag/ingest.py` | CLI: reads `data/rag/*.md`, writes `chunks.jsonl`. |
| `data/rag_pdfs/` | Source PDF directory for legal/tax hard-rules corpus. |
| `data/rag_index/legal_chunks.jsonl` | Audit JSONL from PDF ingestion. Includes `source_document`, `page_number`, `parent_section`, `chunk_type`. |
| `data/rag_vector/` | Persistent Chroma vector index for PDF chunks. |
| `app/rag/ingest_pdf_vector.py` | CLI: parses legal/tax PDFs hierarchically and ingests into Chroma. |
| `app/rag/retrieval_vector.py` | `RAGVectorRetriever`: vector + keyword hybrid ranking with same runtime contract as `RAGRetriever`. |
| `app/rag/retrieval.py` | `RAGRetriever`: loads JSONL, builds a query from the user‚Äôs goal, ranks topic-matching chunks first, then backfills with other topics if needed. |
| `app/rag/prompts.py` | `format_rag_message()`: wraps chunks in a **grounding** system block (no LLM calls). |
| `app/orchestrator.py` | Calls `format_rag_message` + passes `rag_context` into `Speaker.run()`. |
| `app/speaker.py` | Accepts optional `rag_context`; inserts it as an extra `system` message after the orchestrator instruction. |
| `app/streamlit_app.py` | Builds `RAGRetriever`; sidebar shows RAG status. |

Prompt library for **authoring** new RAG content (optional): `prompts/rag_prompt_library.html` / `.json`.

## 4. How retrieval works (design choices)

- **No GPU, no new dependencies:** Scoring uses token overlap between a **query string** and each chunk (`app/rag/retrieval.py`). This is easy to run in class projects and easy to debug.
- **Topic preference:** Chunks whose `topic` equals `state.goal.primary_goal` are ranked first; if fewer than `k` match, the retriever backfills with other topics.
- **Query text:** Built from the primary goal plus a small keyword map (`GOAL_KEYWORDS`) so queries align with each track (credit, 401(k), loans, etc.).
- **Vector option:** `RAG_BACKEND=vector` uses Chroma similarity search with hybrid reranking (vector distance + keyword overlap + topic bonus).

You can later swap this for **embedding** retrieval (same JSONL, different scorer) without changing the orchestrator contract.

## 5. How the chatbot behavior changes for users

| Phase | Change |
|-------|--------|
| 0‚Äì2 | **No change** ‚Äî no RAG injection. |
| **3** | The model receives **retrieved excerpts** as an additional system message to help ground ‚Äúwhere to find numbers‚Äù and term explanations during evidence intake. |
| **4** | The model receives **retrieved excerpts** as an additional system message. The plan should align **when possible** with those excerpts; the grounding text tells the model not to invent specifics absent from the base. |
| 5 | **No change** unless you extend the code to add RAG elsewhere. |

Users do **not** see raw chunk IDs in the UI; excerpts are internal context for the model.

## 6. Configuration (environment variables)

| Variable | Default | Meaning |
|----------|---------|---------|
| `RAG_ENABLED` | `1` | Set to `0` or `false` to disable RAG entirely. |
| `RAG_PROFILE` | _(unset)_ | Set to `fast` for lower cost/latency: retrieve 2 chunks, 2 in prompt, 700 chars/chunk (unless overridden). |
| `RAG_TOP_K` | profile-based | Chunks to retrieve; if unset, `2` when `RAG_PROFILE=fast`, else `3`. |
| `RAG_MAX_CHUNKS_IN_PROMPT` | profile-based | Max chunks in the grounding message; if unset, `2` when `fast`, else `3`. |
| `RAG_MAX_CHARS_PER_CHUNK` | profile-based | Per-chunk excerpt cap; if unset, `700` when `fast`, else `1200`. |
| `RAG_BACKEND` | `jsonl` | `jsonl` for markdown index, `vector` for PDF vector index. |
| `RAG_VECTOR_COLLECTION` | `finlit_hard_rules` | Chroma collection name when vector backend is enabled. |

Add to `.env` as needed, e.g.:

```env
RAG_ENABLED=1
RAG_PROFILE=fast
RAG_BACKEND=jsonl
```

## 7. Rebuilding the index after editing sources

From the **project root**:

```bash
python app/rag/ingest.py
```

This overwrites `data/rag_index/chunks.jsonl`. Restart Streamlit (or rely on `@st.cache_resource` invalidation by restarting the app) so the retriever reloads.

For legal/tax PDFs (hierarchical parsing + vector ingestion):

```bash
python app/rag/ingest_pdf_vector.py
```

This writes:
- `data/rag_index/legal_chunks.jsonl` (auditable chunk dump)
- `data/rag_vector/` (persistent Chroma index)

Expected default PDFs under `data/rag_pdfs/`:
- `IRS Publication 560.pdf`
- `IRS Publication 575.pdf`
- `IRS Publication 590b.pdf`
- `US Code Title 26 Section 401.pdf`

## 8. Safety and scope

- Source texts are written in **general educational** language and avoid product names; **you** remain responsible for accuracy and for aligning with course policy.
- Product-name **guards** in the orchestrator still apply to the **final** assistant text.
- RAG **does not** constitute personalized financial advice; the Speaker skill and disclaimers still apply.

## 9. Troubleshooting

| Symptom | What to check |
|---------|----------------|
| Sidebar says index missing | Run `python app/rag/ingest.py`; confirm `data/rag_index/chunks.jsonl` exists. |
| Sidebar says vector index missing | Set `RAG_BACKEND=vector`, run `python app/rag/ingest_pdf_vector.py`, confirm `data/rag_vector/` exists. |
| RAG always off | Set `RAG_ENABLED=1`; ensure chunks file is non‚Äëempty. |
| Replies ignore excerpts | Model may still generalize; tighten Phase 3/4 speaker skills or add more on‚Äëtopic chunks. |
| Wrong topic chunks | Ensure `data/rag/<topic>.md` filename matches `primary_goal` exactly. |

## 10. Summary

**Implemented:** Dual-backend RAG with (a) keyword JSONL retrieval and (b) vector retrieval over hierarchical legal/tax PDF chunks. Retrieval runs in **Phase 3 and Phase 4**, with optional env toggles and dedicated ingest scripts.

**Effect:** More **grounded** educational content during evidence intake and plan generation, when excerpts match the user‚Äôs selected goal.
