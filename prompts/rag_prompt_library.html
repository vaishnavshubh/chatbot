Prompt library

RAG & content generation — sophisticated prompts

Use these with ChatGPT or similar tools to draft glossaries, JSONL chunks, ingest scaffolding, grounding blocks, and review passes for the financial literacy chatbot. Always fact-check generated content against your course materials. Machine-readable twin: rag_prompt_library.json in this folder.

Global constraints (prepend to any prompt)

  • Educational framing only; no personalized financial, tax, or legal advice.
  • Never recommend specific banks, brokerages, funds, stocks, or credit cards by name.
  • Use hypothetical or illustrative numbers only unless citing user-provided values.
  • Plain English for early-career adults (approx. grades 10–14 reading level).
  • Ask for Markdown or JSONL only when specified; no extra commentary unless requested.
  • Mark uncertain factual claims with [verify] for manual review.

1. Glossary & topic Markdown (data/rag/*.md)

id: glossary_markdown

System / developer role: You are helping build an educational (not advisory) financial literacy glossary for college students and early-career professionals.

Constraints:
- No specific product, bank, fund, or stock names.
- No personalized advice; use "many people" / "in general" framing.
- Short definitions (2–5 sentences), plain English.
- Include a "Common misconceptions" bullet where useful.

Output format: Markdown with ## headings per term.

Topics to cover (one section each): budgeting, emergency fund, credit utilization, APR, employer 401(k) match, vesting, student loan repayment plans at a high level.

For each term, include:
- Definition
- Why it matters
- One hypothetical example (clearly labeled as illustrative)

If a detail varies by employer, loan type, or jurisdiction, say so briefly and add [verify] instead of guessing.

Replace the topic list with your own list (comma-separated). For course alignment, paste a short outline from your syllabus into the prompt as additional constraints.

2. RAG chunks as JSONL (data/rag_index/chunks.jsonl)

id: chunks_jsonl

System / developer role: You produce machine-readable retrieval chunks for a RAG pipeline used in an educational financial literacy chatbot.

I need a JSONL file. Each line must be exactly one JSON object with keys: id, text, source, topic

Rules:
- text: 150–400 words, self-contained, no markdown code fences inside text
- source: synthetic filename like "glossary.md" or "topic_credit.md"
- topic: one of [financial_foundations, budget_cashflow, credit_management, workplace_401k, student_loans, borrowing_basics]
- id: stable slug, e.g. "credit_utilization_001"

Generate 12 chunks total, distributed across topics as evenly as possible.
Educational only; no product recommendations; no account numbers or PII.

Output ONLY JSONL lines (one object per line). No commentary, no markdown fences.

Change 12 to the number of chunks you need. Validate JSONL with a linter before committing.

3. Minimal ingest & retrieval Python (keyword scoring)

id: ingest_python_skeleton

System / developer role: You write small, readable Python for a class project prototype.

Write a minimal Python module that:
- Reads all .md files from a folder path argument
- Splits on ## headings into chunks
- Writes chunks.jsonl with objects: {id, text, source, heading}

- Includes a stub embed_text(s: str) -> list[float] that raises NotImplementedError with a comment on where to plug in an embedding API

Keep under ~180 lines, type hints optional, include if __name__ == "__main__": demo.

Project context: financial literacy RAG; chunks are educational text only.

After generation, replace the stub with your Purdue OpenAI-compatible endpoint for embeddings if you use semantic search.

4. Speaker grounding block (retrieved excerpts only)

id: speaker_grounding_block

System / developer role: You write a reusable instruction block for an LLM that receives retrieved context.

Write a single string suitable for a system or developer message in an OpenAI-style chat API. It must instruct the model to:
- Ground explanations primarily in the provided "Retrieved excerpts" section
- If excerpts do not support a claim, say the knowledge base does not specify it
- Never recommend specific financial products; explain decision criteria instead
- Maintain a supportive, non-judgmental educational tone
- End with a one-line disclaimer that this is educational information, not personalized advice

Do not include placeholders like {{excerpts}} inside the string; instead say "the excerpts provided in the user message" or similar.
Maximum ~120 words.
