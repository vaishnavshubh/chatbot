# Implementation Specification

## Purpose

This document tells a developer (or Cursor) how to turn the configuration artifacts generated by Skill 10 into a running LangGraph TypeScript application. Skill 10 produces the *what* — schemas, phase skills, orchestrator rules. This spec defines the *how* — the technology stack, the file generation order, which artifact feeds into which code file, and the runtime constraints.

Read Skill 10 first. Then read this.

---

## Technology Stack

The code must use **LangGraph's official TypeScript API only** (not Python). The generated code must be **fully compilable TypeScript using ESM** (ECMAScript Modules).

## Constraints

These constraints apply to every file generated. They are non-negotiable.

### Strongly Typed Shared State

All data passed between nodes must be defined in a **single shared state interface**. State must be **initialized once**, updated immutably, and **never contain undefined values** unless explicitly marked optional. Every node must **return the full updated state object**.

```typescript
// Every node signature looks like this:
const analyzerNode = async (state: AgentState): Promise<AgentState> => {
  // ... do work ...
  return { ...state, /* updated fields */ };
};
```

### Configuration-Driven Values

Variable values must be defined in a **config object** (`config.ts`), never hard-coded in node logic. The config is organized as per-component objects so each component (analyzer, speaker, summary) bundles its own model, temperature, retry count, and history settings. Phase-level config and global settings are separate sections.

When generating `config.ts`, use placeholder model names. The behavioral defaults below (temperatures, retry counts, history turns) are sensible starting points. The user adjusts them after generation.

```typescript
const config = {
  provider: "openai",  // or "anthropic" — determines which LangChain wrapper to use

  analyzer: {
    model: "REPLACE_WITH_MODEL",
    fallbackModel: null,
    temperature: 0,
    maxRetries: 2,
    historyTurns: 3,
  },

  speaker: {
    model: "REPLACE_WITH_MODEL",
    fallbackModel: "REPLACE_WITH_MODEL",

    model: "REPLACE_WITH_MODEL",
```

**Where config values come from:**
- `orchestrator_rules.md` → transition confidence, conversation limits, error tolerance thresholds, fallback messages, resumption settings, hooks, entity rotation rules
- `phase_registry.json` → per-phase max turns, auto-advance, allowed targets, conditions
- `state_schema.json` → per-phase entity config (entity name, min/max, rotation rule)
- Implementation defaults → model names, temperatures, retry counts, history turns (these don't appear in the skills or rules files)

### Environment-Driven Secrets

Secrets and environment-specific settings must come from environment variables, never from config or code. Generate a `.env` file with placeholder values:

```
LLM_PROVIDER=openai          # openai | anthropic
LLM_API_KEY=your-key-here    # OPENAI_API_KEY or ANTHROPIC_API_KEY depending on provider
LANGCHAIN_API_KEY=your-key-here
LANGCHAIN_TRACING_V2=true
LANGCHAIN_PROJECT=agent-name
```

Validate at startup that `LLM_API_KEY` is present. Warn (don't crash) if tracing env vars are missing. The startup code should set the appropriate provider-specific environment variable (e.g., `OPENAI_API_KEY` or `ANTHROPIC_API_KEY`) from `LLM_API_KEY` based on the configured `LLM_PROVIDER`.

### LangSmith Tracing

The generated code must enable **LangSmith tracing** for the full LangGraph run, capturing **each node execution** and **each LLM/tool call** as trace spans.

Requirements:

- Use **LangChain's official provider wrappers** so traces are captured automatically (`ChatOpenAI` from `@langchain/openai` or `ChatAnthropic` from `@langchain/anthropic`). Do NOT call raw provider SDKs directly — they bypass LangSmith tracing.
- Tracing is controlled entirely via environment variables. No hard-coded tracing flags.
- The graph execution must pass a `RunnableConfig` to `graph.invoke()` / `graph.stream()` to attach:
  - `tags`: workflow name and node category tags (e.g., `"interview-prep"`, `"analyzer"`, `"speaker"`)
  - `metadata`: topic, runId, and config parameters (model name, turn limits, etc.)
- Every node must add minimal structured trace context (via config metadata/tags) so runs can be filtered in LangSmith by topic, node name, and runId.
- Add a delay before `process.exit()` so batched spans can upload.

```typescript
import { v4 as uuidv4 } from "uuid";

const runId = uuidv4();

const result = await graph.invoke(initialState, {
  tags: ["interview-prep", "production"],
  metadata: {
    topic: "interview-prep-coach",
    runId,
    model: config.analyzer.model,
    globalMaxTurns: config.limits.globalMaxTurns,
  },
});

// Allow LangSmith spans to flush
await new Promise((resolve) => setTimeout(resolve, 2000));
```

## Execution Flow

This is the per-turn pipeline. Every user message flows through this sequence:

```
User message
     │
     ▼
┌────────────────────────────┐
│ Analyzer Prompt Creator    │  reads: state.currentPhase (or state.newPhase on redirect)
│                            │  loads: prompts/analyzer_template.md
│                            │  loads: skills/{phase}/analyzer.md
│                            │  concatenates template + phase skill
│                            │  computes all variable values from state
│                            │  calls: prompt-loader to populate {{placeholders}}
│                            │  outputs: state.analyzerPrompt
└──────────┬─────────────────┘
           │
           ▼
┌────────────────────────────┐
│ Analyzer                   │  reads: state.analyzerPrompt + user message
│                            │  calls: LLM → returns JSON extraction
│                            │  sets: state.analyzerOutput (extraction)
│                            │  sets: state.newPhase (from phase_suggestion or currentPhase)
└──────────┬─────────────────┘
           │
      ┌────┴────────────────┐
      │                     │
  newPhase !==          newPhase ===
  currentPhase          currentPhase
  AND userChanged-        │
  Phase < 2               │ (reset userChangedPhase = 0)
      │                    ▼
      │         ┌────────────────────────────┐
      │         │ State Updater              │  validates extraction against schema
      │         │                            │  merges fields into state (entity-aware)
      │         │                            │  evaluates: stay / advance / complete
      │         │                            │  manages: entity rotation
      │         │                            │  determines: turn type for Speaker
      │         │                            │  sets: transitionDecision, turnType
      │         └──────────┬─────────────────┘
      │                    │
  newPhase !==        ┌────┴──────┐
  currentPhase        │           │
  AND userChanged- complete    stay/advance
  Phase >= 2          │           │
      │               ▼           ▼
      │              END  ┌────────────────────────────┐
      │                   │ Speaker Prompt Creator      │  reads: updated state, turnType
      │                   │                             │  loads: prompts/speaker_template.md
      │                   │                             │  loads: skills/{phase}/speaker.md
      │                   │                             │  concatenates + computes variables
      │                   │                             │  calls: prompt-loader to populate
      │                   │                             │  outputs: state.speakerPrompt
      │                   └──────────┬─────────────────┘
      │                              │
      │                              ▼
      │                   ┌────────────────────────────┐
      │                   │ Speaker                     │  reads: state.speakerPrompt
      │                   │                             │  calls: LLM → returns response text
      │                   │                             │  appends to messages
      │                   └────────────────────────────┘
      │
      └──→ back to Analyzer Prompt Creator
           (APC increments userChangedPhase,
            builds prompt for newPhase)
```

**First-turn exception:** On the first turn, there is no user message. The Analyzer Prompt Creator and Analyzer are skipped. The State Updater sets the default phase from `phase_registry.json`, sets the turn type to `first_turn`, and flow proceeds to Speaker Prompt Creator → Speaker. The Speaker uses its `speaker.md` opening message guidance to generate the conversation's first message.

**Phase redirect routing:** After the Analyzer, a conditional edge compares `state.newPhase` to `state.currentPhase`. If different AND `userChangedPhase < 2`, flow returns to the Analyzer Prompt Creator, which increments `userChangedPhase` and builds the prompt for `newPhase`. If the Analyzer confirms the phase on the second pass (no change), `userChangedPhase` resets to 0 and flow proceeds to the State Updater normally. If `userChangedPhase` reaches 2 and the Analyzer changes phase *again*, the system logs the oscillation, sets `clarificationNeeded` to a rephrase request, and routes to the State Updater which sends the Speaker a clarification turn asking the user to rephrase.

**Post-State-Updater routing:** After the State Updater, a conditional edge checks `transitionDecision`. If `"complete"`, route to END. Otherwise route to Speaker Prompt Creator.

Note: Analyzer retries are internal to the Analyzer node (retry with stricter prompt up to max retries). They do not appear as graph routing decisions. When the State Updater needs clarification from the user, it sets `turnType: "clarification"` and `transitionDecision: "stay"`, routing to the Speaker Prompt Creator which ultimately produces a clarifying question.

**Post-response summarization:** After the Speaker's response is printed to the user, the conversation loop in `index.ts` checks whether summarization is due (every N turns, per config). If so, it calls a summarizer utility that compresses older turns into the cumulative summary. This runs after the user sees the response — it never blocks the user.

**Entity rotation:** Within a turn, the State Updater checks whether the current entity is complete per the rotation rule (from orchestrator_rules.md Entity Rotation Rules, surfaced via config). If complete and more entities are expected (`currentEntityIndex + 1 < maxEntities` and user hasn't signaled done), it advances `currentEntityIndex`, initializes empty per-entity fields for the new entity, and sets `turnType: "entity_transition"`. The Speaker Prompt Creator then assembles the prompt with entity-transition context.

### File 1: `src/state.ts` — Shared State Interface

**Reads from agent_config:**
- `state_schema.json` — field names, types, required/optional, defaults

**Imports from src:** nothing (this is the foundation)

**What it does:**
Defines the single `AgentState` interface that every node receives and returns. Every field from every phase in the state schema becomes a typed property. Also includes the runtime control fields that the pipeline needs.

**Derivation rules:**
- `"string"` in schema → `string` in TypeScript
- `"integer"` in schema → `number` in TypeScript
- `"boolean"` in schema → `boolean` in TypeScript
- `"string[]"` in schema → `string[]` in TypeScript
- `"object[]"` in schema → `Record<string, unknown>[]` in TypeScript (or a specific interface if the object structure is defined)
- `"enum: a | b | c"` in schema → `"a" | "b" | "c"` union type
- Required fields with no default → must be initialized to `null` (use `string | null` type)
- Fields with `"default"` in schema → use that default in the initial state factory

**Must include these runtime control fields** (not from schema — these are pipeline overhead):

```typescript
interface AgentState {
  // --- Runtime control ---
  conversationId: string;
  currentPhase: string;                    // which phase the pipeline is operating in
  newPhase: string;                        // set by Analyzer from phase_suggestion; compared to currentPhase for phase redirect routing
  previousPhase: string | null;
  turnCount: number;
  turnType: "first_turn" | "standard" | "phase_transition" | "clarification" | "entity_transition";
  messages: Array<{ role: "user" | "assistant"; content: string }>;
  analyzerPrompt: string | null;           // assembled by APC, consumed by Analyzer
  analyzerOutput: AnalyzerResult | null;
  speakerPrompt: string | null;            // assembled by SPC, consumed by Speaker
  phaseComplete: Record<string, boolean>;
  transitionDecision: "stay" | "advance" | "complete";
  crossPhaseContext: string;
  priorPhaseSummary: string;
  clarificationNeeded: string | null;
  conversationSummary: string;
  lastSummarizedTurnIndex: number;
  error: string | null;

  // --- Phase redirect tracking ---
  userChangedPhase: number;                  // 0 = no redirect, 1 = first redirect, 2 = second redirect
                                             // If Analyzer changes phase at 2, log problem, ask user to rephrase

  // --- Entity tracking ---
  currentEntityIndex: number;              // 0-based index within current phase's entity array
  entities: Record<string, Array<Record<string, unknown>>>;  // per-phase entity arrays

  // --- Error tracking ---
  errorTracking: {
    consecutiveErrors: number;       // Reset on successful turn, escalate at threshold
    emptyExtractionStreak: number;   // Consecutive turns with no extraction (Skill 8 off-topic detection)
    refusalsPerField: Record<string, number>;  // Per-field refusal count (Skill 8 refusal handling)
    clarificationAttempts: number;   // Attempts to resolve current conflict (Skill 8 contradiction handling)
  };

  // --- Domain fields (derived from state_schema.json) ---
  // Phase-level fields remain flat properties (unchanged)
  // Per-entity fields live in the entities map
}
```

**Field notes:**
- `currentPhase` — the phase the pipeline is currently operating in. Set by the State Updater. Renamed from `activePhase` for clarity alongside `newPhase`.
- `newPhase` — set by the Analyzer from its `phase_suggestion` output field. If `null` or same as `currentPhase`, no phase change detected. If different, triggers phase redirect routing. The State Updater reconciles `newPhase` into `currentPhase` when processing the transition.
- `analyzerPrompt` — the fully assembled prompt string for the Analyzer, built by the Analyzer Prompt Creator. The Analyzer node consumes this directly without needing to know about templates or skill files.
- `speakerPrompt` — the fully assembled prompt string for the Speaker, built by the Speaker Prompt Creator. Same principle.
- `userChangedPhase` — tracks how many phase redirect passes have occurred in the current turn. Reset to 0 by `index.ts` before each `graph.invoke()`. Incremented by the Analyzer Prompt Creator when building a prompt for `newPhase`. If it reaches 2 and the Analyzer changes phase again, the system logs the oscillation and asks the user to rephrase.
- `currentEntityIndex` — 0-based index into the entity array for the current phase. Only meaningful for entity-bearing phases. Reset to 0 on phase transitions.
- `entities` — maps phase names to arrays of entity records. Each entity is a `Record<string, unknown>` containing per-entity field values. Initialized to `{}` and populated as entity-bearing phases are entered.
- `crossPhaseContext` — structured field data carried from completed phases to the current phase's Speaker (e.g., "Company: Google, Format: system design"). Built by the State Updater from the cross-phase context config.
- `priorPhaseSummary` — a narrative bridge sentence generated on phase transition (e.g., "Now that we've identified your target company and role, let's move on to practice."). Used by the Speaker for the `phase_transition` turn type.
- `conversationSummary` — the cumulative summary of older turns, updated periodically by the summarizer utility. Starts as empty string.
- `lastSummarizedTurnIndex` — the index into `messages` up to which the summary covers. Starts at 0.

**Turn type values** (defined by Skill 3 — Speaker Prompt Engineering):
- `first_turn` — First turn of a phase with no user message to respond to
- `standard` — Normal turn where the user has sent a message
- `phase_transition` — A transition just occurred; Speaker bridges to the new phase
- `clarification` — State Updater detected a conflict or ambiguity needing user input
- `entity_transition` — State Updater decided to rotate to the next entity within the current phase; Speaker should close out the previous entity and open the next (see Skill 3's Entity Transition Turn Type)

**Must also export:**
- `createInitialState()` factory function that returns a fully initialized `AgentState` with all defaults set, `turnType` set to `"first_turn"`, `currentPhase` set to `""` (set by State Updater on first turn), `newPhase` set to `""`, `userChangedPhase` set to `0`, `currentEntityIndex` set to `0`, `entities` set to `{}`, `analyzerPrompt` set to `null`, `speakerPrompt` set to `null`, `conversationSummary` set to `""`, `lastSummarizedTurnIndex` set to `0`, `errorTracking` initialized to `{ consecutiveErrors: 0, emptyExtractionStreak: 0, refusalsPerField: {}, clarificationAttempts: 0 }`, and no `undefined` values
- `AnalyzerResult` interface for the structured JSON the Analyzer returns

### File 2: `src/config.ts` — Configuration Object

**Reads from agent_config:**
- `orchestrator_rules.md` — confidence thresholds, error tolerance, conversation limits, fallback messages, resumption, hooks
- `phase_registry.json` — max turns per phase, auto-advance flags

**Imports from src:** nothing

**What it does:**
Exports a typed `AgentConfig` object containing every variable value. Node logic references `config.analyzer.model`, `config.limits.globalMaxTurns`, etc. — never magic numbers. Also exports a helper function that creates the appropriate LangChain chat model based on `config.provider`.

**Must include:**
- `provider` — `"openai"` or `"anthropic"`
- `analyzer` — `{ model, fallbackModel, temperature, maxRetries, historyTurns }`
- `speaker` — `{ model, fallbackModel, temperature, maxRetries, historyTurns }`
- `summary` — `{ model, fallbackModel, temperature, summarizeEveryNTurns, summarizeOnPhaseTransition }`
- `limits` — globalMaxTurns, conversationTimeoutSeconds
- `transitions` — minConfidence threshold
- `errorTolerance` — consecutiveErrorThreshold, escalationAction
- `phaseRedirect` — `{ maxAttempts }`. Controls how many times the Analyzer can redirect to a different phase in a single turn before asking the user to rephrase. Default `2`. No per-phase configuration needed — the behavior is automatic and universal.
- `entityDefaults` — `{ maxEntities }`. Global default overridden per-phase.
- `fallbackMessages` — one message per turn type (`first_turn`, `standard`, `phase_transition`, `clarification`, `entity_transition`) plus `termination`. Parsed from `orchestrator_rules.md` Fallback Messages section. The Speaker node uses these when it fails after all retries.
- `resumption` — `{ enabled, ttlSeconds }`. Parsed from `orchestrator_rules.md` Resumption section.
- `hooks` — `{ preConversation, midPipeline, postCompletion, preResumption }`. Each is either `null` (no hook) or an async function. Parsed from `orchestrator_rules.md` Hooks section. Hooks are extension points — the generated code provides the call sites; domain authors implement the functions.
- `phases` — per-phase config derived from `phase_registry.json` and `state_schema.json`: `maxTurns`, `autoAdvance`, `allowedTargets`, `conditions` (from registry), `entityConfig` (`null` or `{ entityName, displayName, minEntities, maxEntities, rotationRule }`, from schema)

**Must also export:**
- `createChatModel(componentConfig)` — factory that returns `ChatOpenAI` or `ChatAnthropic` based on `config.provider`, using the component's model and temperature. This is the single place where the provider choice is resolved.

### File 3: `src/utils/prompt-loader.ts` — Prompt Template Engine

**Reads from agent_config:** nothing (files loaded by Prompt Creator nodes)

**Imports from src:** nothing (pure utility)

**What it does:**
A thin template engine. Receives a concatenated prompt string and a variables map. Performs `{{placeholder}}` replacement. Validates no unresolved placeholders remain. This file has no knowledge of which templates exist, what variables they expect, or how to compute variable values — that intelligence lives in the Prompt Creator nodes.

**Must export:**

```typescript
// Core template population
export function populateTemplate(
  template: string,
  variables: Record<string, string>
): string;
// Replaces every {{key}} in template with the corresponding value from variables.
// Throws if any {{placeholder}} remains unresolved after replacement.
// Throws if a key in variables doesn't match any placeholder (catches typos).

// File loading helper (both Prompt Creators need this)
export function loadFile(relativePath: string): string;
// Reads a file from the agent_config directory. Returns the full file contents.
// Throws if the file doesn't exist (structural problem, caught by smoke test).
```

**What moved to Prompt Creator nodes:**
The previous version of this file contained prompt assembly logic (`assembleAnalyzerPrompt()`, `assembleSpeakerPrompt()`, `assembleSummaryPrompt()`), data formatting functions (`formatCollectedData()`, `formatMissingFields()`, `formatClarificationNeeded()`), and phase registry summary generation (`generatePhaseRegistrySummary()`). All of this logic now lives in the Prompt Creator nodes (#6 and #9) and the summarizer (#5), which own the intelligence of *what* to inject. This file is now a pure template engine that only knows *how* to inject.

### File 4: `src/utils/history-manager.ts` — Conversation History

**Reads from agent_config:** nothing (uses config values passed in)

**Imports from src:**
- `config.ts` — for per-component history turns and truncation limits
- `state.ts` — for message types

**What it does:**
Assembles conversation history for inclusion in prompts. Combines the cumulative summary with recent unsummarized turns (see Skill 7 — Conversation History Management). All components receive the same history format — a simple `[User]`/`[Assistant]` transcript.

**Key behavior from Skill 7:**
- History = [cumulative summary] + [recent unsummarized turns]. The summary covers older turns; recent turns are included in full.
- All components (Analyzer, Speaker, Summarizer) receive the same history format. No per-component formatting — the format is a uniform `[User]`/`[Assistant]` labeled transcript.
- The current user message in `{{user_message}}` is never truncated — only messages in the history section.

**Must export:**
- `getConversationSummary(conversationSummary)` → `string` — Returns the cumulative summary string for injection into `{{conversation_summary}}`. Returns empty string if no summary exists yet.
- `getRecentTurns(messages, lastSummarizedTurnIndex, historyTurns)` → `string` — Returns recent unsummarized turns formatted as a `[User]`/`[Assistant]` transcript for injection into `{{recent_turns}}`. These are the turns since the last summarization point, capped to `historyTurns`.
- `shouldSummarize(turnCount, summarizeEveryNTurns)` → `boolean` — Returns true when it's time to run summarization
- `getUnsummarizedTurns(messages, lastSummarizedTurnIndex)` → `string` — Returns all unsummarized turns as a simple transcript for the summary prompt (not capped — the summarizer needs the full set)

**Important:** The Analyzer template (Skill 2) and Speaker template (Skill 3) inject `{{conversation_summary}}` and `{{recent_turns}}` as **separate** placeholders. The prompt-loader must receive these as two distinct values, not a single combined string. This is why `getConversationSummary()` and `getRecentTurns()` are separate functions.

### File 5: `src/utils/summarizer.ts` — Post-Response Summarization

### File 8: `src/nodes/state-updater.ts` — State Updater Node (Replaces orchestrator.ts)

**Reads from agent_config:**
- `state_schema.json` — field types, required flags, update policies, validation rules, entity config
- `phase_registry.json` — allowed transitions, conditions, auto-advance
- `orchestrator_rules.md` — business rules, cross-phase context, default flow, entity rotation rules

**Imports from src:**
- `state.ts` — `AgentState`
- `config.ts` — phase configs, transition thresholds, limits

**What it does:**
This is the deterministic brain of the pipeline — no LLM calls. It receives state with `analyzerOutput` populated by the Analyzer (already analyzed against the correct phase, since phase redirect happened if needed).

Steps:

0. **Snapshot state** before any mutation. Deep-copy the incoming state object. If any integrity check fails after mutation (step 13), roll back to this snapshot and return it with an error logged. See Skill 8's State Integrity section.
1. **Check for phase oscillation**: if `newPhase !== currentPhase` AND `userChangedPhase >= config.phaseRedirect.maxAttempts`, the Analyzer couldn't settle on a phase. Log a warning ("Analyzer oscillated between phases after N redirects"). Set `clarificationNeeded` to a natural-language rephrase request (e.g., "I wasn't sure what you meant — could you rephrase that?"). Set `turnType: "clarification"`, `transitionDecision: "stay"`, `userChangedPhase: 0`. Skip steps 2–7 and proceed to step 8.
2. **Commit phase change** if `newPhase !== currentPhase`: set `currentPhase = newPhase`, set `previousPhase` to old phase, update `phaseComplete` for old phase, reset entity tracking for the new phase (`currentEntityIndex = 0`, initialize entity array).
3. **Validate** the Analyzer's JSON output against the phase's field schema (strip unknown fields, check types, reject hallucinated fields).
4. **Merge** valid extracted fields into state following update policies from `state_schema.json`:
   - Flat phases: `overwrite` replaces value, `append` merges into array with deduplication, `conflict` flags for clarification (same as before)
   - Entity phases: per-entity fields → `entities[currentPhase][currentEntityIndex]`, phase-level fields → flat state properties
5. **Entity rotation check** (for entity-bearing phases): evaluate rotation rule from config. If the current entity's required per-entity fields are all filled and the rotation trigger is met, increment `currentEntityIndex`, initialize a new empty entity in the array, set `turnType: "entity_transition"`. If `currentEntityIndex >= maxEntities`, do not rotate — check phase completion instead.
6. **Check phase completion**: for flat phases — all required fields non-null. For entity phases — `min_entities` met AND all per-entity required fields filled for the current entity AND phase-level required fields filled.
7. **Evaluate transitions**: consult `phase_registry.json` for allowed targets and conditions. Check business rules from `orchestrator_rules.md`. Decide: stay, advance, or complete.
8. **Determine turn type** for the Speaker:
   - `first_turn` — when `turnType` is already `"first_turn"` (opening message)
   - `standard` — normal turn where the user sent a message and the Analyzer processed it
   - `phase_transition` — a transition was just decided; the Speaker needs to bridge to the new phase
   - `clarification` — a conflict was detected (field update policy is `conflict`), the Analyzer's confidence was below the threshold, or the phase oscillation check in step 1 set it. Sets `transitionDecision: "stay"`.
   - `entity_transition` — set in step 5 when entity rotation occurs
9. **Assemble cross-phase context**: when advancing, build `crossPhaseContext` from data collected in completed phases per the cross-phase context config in `orchestrator_rules.md`. Format field values using `display_name` from the state schema. This maps to `{{cross_phase_context}}` in the Speaker template.
10. **Assemble prior phase summary**: on phase transition turns only, build `priorPhaseSummary` as a brief narrative bridge sentence for the Speaker's `phase_transition` turn. This is used within `{{turn_type_instructions}}` for `phase_transition` turns.
11. **Assemble clarification context**: when routing to a clarification turn, format the conflicting or ambiguous fields as natural-language questions (not field names) using `description` from the state schema.
12. **Enforce limits**: check turn count against per-phase and global max turns.
13. **Verify state integrity** after mutation (see Skill 8): active phase exists in registry, field types match schema, completion flags match actual data, turn counts are sane, entity array lengths are consistent. If any check fails, roll back to the snapshot from step 0.
14. **Update error tracking**: if the Analyzer produced a valid extraction, reset `errorTracking.consecutiveErrors` and `errorTracking.emptyExtractionStreak`. If the extraction was empty, increment `emptyExtractionStreak`. If the streak exceeds the off-topic threshold (default 3), signal the Speaker to redirect. If a phase transition occurs, reset `emptyExtractionStreak`, `refusalsPerField`, and `clarificationAttempts`. Also reset `userChangedPhase` to 0 (consumed for this turn).
15. **Run mid-pipeline hook**: if `config.hooks.midPipeline` is configured, call it after state is updated but before returning. If the hook fails, log and proceed — mid-pipeline hooks must not block the Speaker (see Skill 6).

**Must set on state:**
- `transitionDecision`: `"stay"` | `"advance"` | `"complete"`
- `turnType`: one of the five turn types, determined by the logic above
- `currentPhase`: updated if advancing or committing a phase change
- `previousPhase`: set to old phase if advancing
- `phaseComplete`: updated completion flags
- `crossPhaseContext`: structured field data from prior phases, formatted with `display_name` (updated on advance, otherwise preserved)
- `priorPhaseSummary`: narrative bridge text (set only on phase transitions)
- `clarificationNeeded`: natural-language clarification question if routing to clarification, null otherwise
- `currentEntityIndex`: updated on entity rotation
- `entities`: updated entity arrays
- `userChangedPhase`: reset to 0
- `errorTracking`: updated per step 14
- Phase data fields: merged from analyzer output

**Node signature:**

### File 12: `src/index.ts` — Entry Point

**Reads from agent_config:** nothing (all access goes through other files)

**Imports from src:**
- `graph.ts` — compiled graph, `createInitialState()`
- `config.ts` — for provider, metadata in tracing
- `utils/summarizer.ts` — `maybeSummarize()`

**What it does:**
The entry point that runs the conversation loop. Validates environment variables based on the configured provider, creates initial state, handles the first-turn (no user message) flow, then loops: read user input → invoke graph → print response → maybe summarize → repeat until complete.

**Must handle:**
1. Provider-dependent environment variable validation: if `config.provider` is `"openai"`, crash on missing `OPENAI_API_KEY`; if `"anthropic"`, crash on missing `ANTHROPIC_API_KEY`. Warn (don't crash) if tracing env vars are missing.
2. **Pre-conversation hook**: if `config.hooks.preConversation` is set, call it before the first turn. This is the extension point for loading saved state (resumption), logging, or setup.
3. First-turn invocation (no user message — the Speaker generates the opening)
4. Conversation loop with user input
5. **Per-turn state reset**: before each `graph.invoke()`, reset `userChangedPhase` to `0` and `newPhase` to `currentPhase` so the phase redirect loop starts fresh each turn
6. Post-response summarization: after printing the response, call `maybeSummarize(state)` to update the cumulative summary if it's time (by turn count or phase transition)
7. **Consecutive error escalation**: after each turn, check `state.errorTracking.consecutiveErrors` against `config.errorTolerance.consecutiveErrorThreshold`. If exceeded, execute `config.errorTolerance.escalationAction` (`"terminate"` ends the conversation gracefully).
8. Passing `RunnableConfig` with tags and metadata to every `graph.invoke()` call
9. **Post-completion hook**: if `config.hooks.postCompletion` is set, call it after the conversation loop ends. This is the extension point for saving final state, sending data downstream, or cleanup.
10. Graceful exit with delay for LangSmith span upload

**Structure:**

```typescript
import { graph, createInitialState } from "./graph.js";
import { config } from "./config.js";
import { maybeSummarize } from "./utils/summarizer.js";
import { v4 as uuidv4 } from "uuid";
import * as readline from "readline";

// 1. Validate env vars (provider-dependent)
const apiKeyVar = config.provider === "openai" ? "OPENAI_API_KEY" : "ANTHROPIC_API_KEY";
if (!process.env[apiKeyVar]) {
  throw new Error(`${apiKeyVar} is required for provider "${config.provider}"`);
}

const runId = uuidv4();
const runnableConfig = {
  tags: ["agent-name", "production"],
  metadata: {
    runId,
    model: config.analyzer.model,
    globalMaxTurns: config.limits.globalMaxTurns,
  },
};

// 2. Pre-conversation hook
let state = createInitialState();
if (config.hooks.preConversation) {
  state = await config.hooks.preConversation(state);
}

// 3. First turn — generate opening message
state = await graph.invoke(state, runnableConfig);
console.log(`Assistant: ${state.messages.at(-1)?.content}`);

// 4. Conversation loop
const rl = readline.createInterface({ input: process.stdin, output: process.stdout });

while (state.transitionDecision !== "complete") {
  const userInput = await new Promise<string>((resolve) => {
    rl.question("You: ", resolve);
  });

  // 5. Per-turn state reset
  state = {
    ...state,
    messages: [...state.messages, { role: "user", content: userInput }],
    userChangedPhase: 0,
    newPhase: state.currentPhase,
  };

  state = await graph.invoke(state, runnableConfig);
  console.log(`Assistant: ${state.messages.at(-1)?.content}`);

  // 6. Post-response summarization (runs after user sees response)
  state = await maybeSummarize(state);

  // 7. Consecutive error escalation
  if (state.errorTracking.consecutiveErrors >= config.errorTolerance.consecutiveErrorThreshold) {
    console.log("Too many consecutive errors. Ending conversation.");
    break;
  }
}

// 8. Post-completion hook
if (config.hooks.postCompletion) {
  await config.hooks.postCompletion(state);
}

// 9. Cleanup
rl.close();
await new Promise((resolve) => setTimeout(resolve, 2000)); // flush LangSmith spans
process.exit(0);
```

If any step fails, there is a structural problem in the generated artifacts.

## Prompt Template Placeholder Reference

The prompt templates (`prompts/analyzer_template.md`, `prompts/speaker_template.md`, and `prompts/summary_template.md`) are the **source of truth** for which `{{placeholder}}` variables exist. The **Prompt Creator nodes** are responsible for computing values for every placeholder their respective template contains.

- `analyzer-prompt-creator.ts` reads `prompts/analyzer_template.md`, identifies all `{{placeholder}}` patterns, and computes a value for each from state and config. See Skill 2 for placeholder definitions and formatting requirements.
- `speaker-prompt-creator.ts` reads `prompts/speaker_template.md`, identifies all `{{placeholder}}` patterns, and computes a value for each from state and config. See Skill 3 for placeholder definitions and formatting requirements.
- `summarizer.ts` reads `prompts/summary_template.md` and computes `{{conversation_summary}}` and `{{recent_turns}}`. See Skill 7.

The `populateTemplate()` utility in `prompt-loader.ts` validates that no unresolved placeholders remain after injection. If a template is updated with new placeholders, the Prompt Creator will fail at runtime until the variable map is updated — surfacing the gap immediately rather than silently leaving an unresolved `{{variable}}` in the prompt.

**Key formatting rules** (from Skills 2 and 3):
- Analyzer state data (`{{active_phase_state_json}}`): raw JSON including nulls. For entity phases: current entity + phase-level fields + prior entities summary (see Skill 2 Variable Formatting Notes).
- Speaker collected data (`{{phase_collected_data}}`): natural key-value pairs using `display_name`. For entity phases: current entity + brief prior entity references (see Skill 3 Entity-Bearing Phases formatting).
- Speaker missing fields (`{{phase_missing_required}}`, `{{phase_missing_optional}}`): use `description`, not field names. For entity phases: current entity only.
- Speaker clarification context (`{{clarification_needed}}`): natural-language questions.
- Speaker entity transition context (`{{turn_type_instructions}}` for `entity_transition`): prior entity names, current entity number, max entities.

## Error Handling Requirements

Each node must handle its own failures gracefully. The pipeline should never crash on a single bad turn. See Skill 8 (Error Recovery and Graceful Degradation) for the full error catalog, recovery strategies, and escalation ladder.

**Analyzer Prompt Creator / Speaker Prompt Creator failures:**
- File not found (missing template or skill file) → crash immediately. This is a structural problem that should be caught by the smoke test before reaching users. It means the agent_config is incomplete.
- Unresolved placeholder after `populateTemplate()` → crash immediately. The variable map is incomplete — the Prompt Creator is missing a variable computation. Also caught by smoke test.

**Analyzer failures:**
- Malformed JSON response → retry with stricter prompt (up to `config.analyzer.maxRetries`). If all fail, set `analyzerOutput` to null, set `error` with reason. State Updater will proceed with empty delta. Increment `errorTracking.consecutiveErrors`.
- LLM timeout or rate limit → retry with exponential backoff (up to `config.analyzer.maxRetries`), then set error. Increment `errorTracking.consecutiveErrors`.
- Hallucinated fields (not in schema) → stripped silently in State Updater, logged as warning.

**State Updater failures:**
- Invalid transition target → reject, stay in current phase, log error.
- State corruption → roll back to pre-mutation snapshot (see State Updater step 0). Verify state integrity after every mutation (step 12 and Skill 8's State Integrity section). Increment `errorTracking.consecutiveErrors`.
- Entity array inconsistency (e.g., `currentEntityIndex` exceeds array length) → roll back to snapshot, log error.
- Max turns exceeded → force transition to terminal or set `transitionDecision: "complete"`.

**Speaker failures:**
- Empty output → retry once. If still empty, use `config.fallbackMessages[state.turnType]` (including `entity_transition` fallback). Increment `errorTracking.consecutiveErrors`.
- LLM timeout → retry (up to `config.speaker.maxRetries`), then use `config.fallbackMessages[state.turnType]`. Increment `errorTracking.consecutiveErrors`.
- Response references internal field names or phase names → see Speaker node Output Validation section. Attempt in-place fix or retry with explicit reminder. Log for investigation.

**General:**
- All errors are logged with: turn number, active phase, node name, error type, and message.
- No error should produce an `undefined` value in state — use `null` explicitly.
- Consecutive error tracking uses `state.errorTracking.consecutiveErrors`: reset to 0 on any successful turn (Analyzer extracted valid data AND Speaker produced clean output). Escalate when the count reaches `config.errorTolerance.consecutiveErrorThreshold` — the `index.ts` loop checks this after each turn and executes `config.errorTolerance.escalationAction`.
- Fallback messages must match the agent's tone. Use the domain-specific messages from `config.fallbackMessages` (parsed from `orchestrator_rules.md` Fallback Messages section), not generic defaults.
